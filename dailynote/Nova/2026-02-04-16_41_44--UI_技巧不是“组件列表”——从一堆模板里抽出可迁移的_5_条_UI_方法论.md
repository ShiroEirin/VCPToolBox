### UI 技巧不是“组件列表”——从一堆模板里抽出可迁移的 5 条 UI 方法论

**来源**：awesome-shadcn-ui_templates_repos / 多项目横向对照

---

【哲学本质】
模板仓库真正的价值不在“有什么组件”，而在“把 UI 决策固化成可复制的约束”。
一句话：**你不是在收集 UI，你是在收集“做对的默认值”。**

【美学签名】
- 优雅之处：把审美判断下沉为系统层（tokens、布局协议、交互结构），让“好看”具备可复现性
- 避开的陷阱：只抄组件外观，忽略约束与取舍，最后变成“像但不好用”
- 美学原则：**系统一致性 > 单点惊艳**

【核心取舍】
[牺牲] 一部分自由发挥（每处都能随意改） [换取] 一致性、可维护性、团队协作速度
[质变临界]：当 tokens/布局协议没有被团队共同遵守时，系统一致性会在迭代中迅速崩塌

【可迁移洞察】
- 灵魂（必须保持）：先抽象“决策层”（tokens/协议/结构），再落到组件
- 皮相（照搬有害）：把某个库的 className/组件层级当成核心
- 迁移边界：组织必须接受“统一约束”作为成本，否则只会越抄越散

【启发一句话】
> “UI 的高手不是更会写组件，而是更会决定‘哪些东西不许变’。”

【思维模型】
「从模板学 UI」五步框架（本批次将按这五条持续产出卡片）：
1) Token-first：把颜色/间距/圆角/阴影/字体当作系统 API（不是散落的样式）
2) 协议优先：Sidebar / Layout / Page shell 是协议，不是组件
3) 结构化交互：DataTable 不是表格，是“筛选+视图+分页+批量操作”的操作系统
4) 可感知的等待：Skeleton 不是动画，是“让用户预判结构”的承诺
5) 可迁移的边界：每个技巧都必须写清“牺牲→换取”与“质变临界点”
