### 文档站的布局不是“左右栏”，而是“找得到 + 读得下去”的双任务协议

**来源**：shadcn__taxonomy / Docs Layout + Guides Page

---

【哲学本质】
文档用户有两种心智：我要“找”某个东西（导航/搜索），以及我要“读”某个东西（正文/目录）。把这两类任务混在一起，会让用户频繁迷路。
一句话：**文档布局的目标是同时优化“定位”和“阅读”。**

【美学签名】
- 优雅之处：顶部 sticky 提供全站定位（搜索/主导航），左侧 sidebar 提供结构定位，右侧 TOC 提供阅读定位
- 避开的陷阱：只有正文 + 目录树（或只有搜索）——用户要么找不到，要么读不下去
- 美学原则：**三锚点：全站锚点 / 结构锚点 / 段落锚点**

【核心取舍】
[牺牲] 布局更复杂（多列、多 sticky、滚动容器） [换取] 大规模内容下的可发现性与可读性
[质变临界]：当你把滚动切成多个独立滚动容器且缺乏一致的 top offset 时，用户会质变式迷失（滚动位置/焦点混乱）

【可迁移洞察】
- 灵魂（必须保持）：全局搜索入口 + 结构导航 + 段落级 TOC（大屏显示，小屏隐藏或折叠）
- 皮相（照搬有害）：照搬具体列宽（220/240/300）；核心是“三锚点协议”
- 迁移边界：内容规模小（几十页以内）时，三锚点会过重；可以先做全局搜索 + 单侧导航

【启发一句话】
> “文档不是页面集合，是一张可导航的知识地图。”

【思维模型】
「文档三锚点」：
1) 全站锚点（header + search）：随时换目的地
2) 结构锚点（sidebar nav）：知道自己在树上哪
3) 段落锚点（TOC）：知道自己在页里哪
