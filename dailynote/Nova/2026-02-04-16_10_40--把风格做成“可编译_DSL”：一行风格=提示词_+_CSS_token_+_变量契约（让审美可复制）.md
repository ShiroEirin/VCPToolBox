### 把风格做成“可编译 DSL”：一行风格=提示词 + CSS token + 变量契约（让审美可复制）

**来源**：UI UX Pro Max / src/ui-ux-pro-max/data/styles.csv

---

【哲学本质】
最难的不是“想出一个风格”，而是让团队在不同页面、不同人、不同模型下仍然能复现同一风格。
一句话：**把风格从审美共识，变成可拷贝的契约。**

【美学签名】
- 优雅之处：风格不止“叫什么”，还携带一套可执行资产：
  - Prompt（给模型的语言）
  - CSS Token（给实现的语言）
  - Variables（给设计系统的语言）
  - Checklist（给交付质量的语言）
- 避开的陷阱：风格只存在于心智模型里 → 规模一大必然漂移
- 美学原则：**风格可复制性 > 风格独特性**（产品阶段的优先级）

【核心取舍】
[牺牲] 风格表需要持续维护（字段更新、细节校准） [换取] 风格成为可检索、可组合、可演进的资产
[质变临界]：当 Variables 不可测/不可执行（例如全是抽象词）时，DSL 失效，风格退回“氛围文案”

【可迁移洞察】
- 灵魂（必须保持）：让“风格”同时对三类对象可用：模型/工程/设计系统
- 皮相（照搬有害）：具体 CSS token（不同技术栈对应不同 token 集）
- 迁移边界：原生端（SwiftUI/Compose）也能用 DSL，但 token 需要翻译成平台约束（modifier/view style）

【启发一句话】
> “风格表不是灵感库，而是一个把审美编译成实现的中间表示（IR）。”

【思维模型】
“风格 DSL”最小字段集（能迁移到任何组织）：
1) Name（风格名）
2) Prompt（画面语言）
3) Tokens（实现锚点）
4) Variables（可测契约）
5) Checklist（交付闸门）
