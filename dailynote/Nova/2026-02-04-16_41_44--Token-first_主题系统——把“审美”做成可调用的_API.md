### Token-first 主题系统——把“审美”做成可调用的 API

**来源**：bundui__shadcn-admin / shadcn__taxonomy / Kiranism__next-shadcn-dashboard-starter

---

【哲学本质】
真正可维护的 UI 不是“到处调 class”，而是把审美决策抽象成一组稳定的变量契约（颜色、半径、阴影、字体、侧栏）。
一句话：**视觉不是实现细节，而是产品 API。**

【美学签名】
- 优雅之处：把颜色语义（background/foreground/sidebar/primary…）从具体值剥离，允许“换皮不换骨”
- 避开的陷阱：用组件库堆出来的“局部漂亮”，在跨页面/跨模块时迅速失去一致性
- 美学原则：**语义优先于数值，系统优先于局部**

【核心取舍】
[牺牲] 立刻随手改出“独特效果”的快感 [换取] 全局一致性、主题可扩展、多人协作时不互相污染
[质变临界]：当团队绕过 tokens 直接写具体颜色/阴影时，主题系统会在量变中失效并回到“样式债务”

【可迁移洞察】
- 灵魂（必须保持）：语义 tokens（不是色值）+ 明暗两套契约 + 组件/布局都只消费 tokens
- 皮相（照搬有害）：照搬某套 palette 或某种色彩空间；关键是“语义层”和“单一出口”
- 迁移边界：需要制定“禁止直写”的团队约束（lint/约定/代码评审），否则只是一份摆设

【启发一句话】
> “主题系统不是让你更容易换颜色，而是让你更难写出不一致的颜色。”

【思维模型】
「Token-first」三层契约：
1) 语义层：background/foreground/sidebar/primary 等——表达意图
2) 映射层：light/dark 或 brandA/brandB——决定风格
3) 消费层：组件/布局只允许从语义层取值——禁止直写数值
