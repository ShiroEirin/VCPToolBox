### 字体不是资源，是系统开关——把“字体选择”从页面里上移到 Token 层

**来源**：Kiranism__next-shadcn-dashboard-starter / shadcn__taxonomy / bundui__shadcn-admin

---

【哲学本质】
在大项目里，字体不只是“好不好看”，而是跨页面一致的语气与密度控制器；如果字体选择散落在组件里，你会得到不可控的“口音污染”。
一句话：**字体要像颜色一样被治理：用 token 统一入口，用主题包统一切换。**

【美学签名】
- 优雅之处：把字体加载（next/font）与字体选择（--font-* tokens）解耦：先准备“字体调色盘”，再在主题里决定用哪一支
- 避开的陷阱：每个页面各自引入不同字体/权重，导致性能、风格、可读性全线漂移
- 美学原则：**单一入口（single entry）+ 可编排（composable）**

【核心取舍】
[牺牲] 初期需要搭建 font variables + theme tokens 的结构 [换取] 大规模一致性、可控的品牌语气、可回滚的风格试验
[质变临界]：当团队开始“临时为某页面换字体”来解决局部问题时，字体系统会迅速碎裂成无法维护的特例集合

【可迁移洞察】
- 灵魂（必须保持）：字体加载层（可选多套）与字体消费层（只消费 --font-sans/mono/serif）分离
- 皮相（照搬有害）：照搬具体字体名（Geist/Plus Jakarta Sans）；核心是“治理结构”
- 迁移边界：如果你没有主题系统/没有 token 约束，字体治理会退化为“约定式自律”，在大团队必然失效

【启发一句话】
> “字体系统的目的不是让你能换字体，而是让你很难随便换字体。”

【思维模型】
「字体治理三层」：
1) 供应层：预加载多套字体（variables）
2) 决策层：主题包选择 sans/mono/serif 的组合（tokens）
3) 消费层：组件只用语义字体（font-sans/font-mono），禁止直接引用具体字体名
